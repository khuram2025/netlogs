# ZenAI Appliance Team — Push & Pull Guide

> **Audience:** The team that builds the ZenAI appliance (both the update agent that runs on the device, and the release package that gets shipped to the fleet).
> **Server:** `https://zentryc.com`
> **Your push endpoint:** `POST https://zentryc.com/api/v1/admin/releases/create` with `-F "product=zenai"`
> **Your dashboard view:** `https://zentryc.com/ota/zenai/releases/`
> **Your appliance pull endpoints:** `/api/v1/appliances/register`, `/checkin`, `/updates/check`, `/updates/download/<id>`, `/updates/report`
> **Document version:** 1.0 (2026-04-17)

This is everything you need to ship a ZenAI release and have a fleet of ZenAI appliances pick it up, apply it, and report back. The server side is already wired — you don't need to coordinate with the OTA server team to get started.

---

## 0. TL;DR

```
PUSH (you, from a build server)
  1. Build a .zup   →   ./build-zenai-release.sh 1.0.0
  2. Sign it        →   (the script signs with your Ed25519 private key)
  3. Upload         →   POST /api/v1/admin/releases/create  -F "product=zenai"
  4. Publish        →   POST /api/v1/admin/releases/<id>/publish
  5. Roll out       →   POST /api/v1/admin/rollouts  {"stage":"canary", ...}

PULL (the appliance agent, every 15 min)
  1. Register once  →   POST /api/v1/appliances/register  (uses a registration token)
  2. Check in       →   POST /api/v1/appliances/checkin   → {"next_action":"update", "release":{...}}
  3. Download       →   GET  /api/v1/updates/download/<id>
  4. Verify         →   sha256 + Ed25519 signature against your embedded public key
  5. Apply          →   run manifest.json steps[] with rollback on failure
  6. Report         →   POST /api/v1/updates/report
```

You only ever touch `product=zenai` releases. The server enforces that ZenAI appliances (subscriptions where `type == "zenai"`) never see ZenPlus releases, and vice-versa. You cannot accidentally push into the wrong channel.

## 1. Before you start — what we give you, what you give us

Items to collect before you write a single line of code:

| What | Who provides | Needed for |
|---|---|---|
| **Admin login** (`email` + password) for `https://zentryc.com` with permission to create `product=zenai` releases | OTA admin | Push flow (§3) |
| **Registration token** per appliance (48-char hex) tied to a ZenAI subscription | OTA admin, via dashboard `Fleet → Subscriptions → Generate keys` | Pull flow, registration (§4.2) |
| **Your ZenAI Ed25519 keypair** (generated by you, per `ZENAI_RELEASES.md` §3) | **You** | Signing builds (§3.3) + verifying on the appliance (§4.5) |
| **ZenAI public key** (`zentryc-zenai.pub`) handed to the OTA admin | **You** → OTA admin | Optional server-side verification (not yet enforced; see §6) |
| **Outbound HTTPS** from every appliance to `zentryc.com:443` | Appliance customer's network | Pull flow (§4) |

What you ship back to us (so we know the channel is live):

- One successful canary release push + apply + report, end-to-end, observed in `/ota/zenai/releases/<id>/`.
- The embedded public key's fingerprint — so we can publish it on our side.

## 3. PUSH — how to ship a ZenAI release

### 3.1 One-time: set up the build workstation

You need:

- Ubuntu 22.04+, Python 3.11+, `pip install cryptography`
- Your private key at `/secure/zentryc-zenai.key`, mode `0600`, owned by the build user
- The build script at `Doc/ZENAI/build-zenai-release.sh` (ready-to-run; this repo already has it)

Environment variables for each push session:

```bash
export ZENTRYC_API="https://zentryc.com/api/v1"
export ZENTRYC_PRIVATE_KEY="/secure/zentryc-zenai.key"
export ZENTRYC_ADMIN_EMAIL="your-zenai-release@email"    # whatever OTA admin gave you
export REPO_DIR="/path/to/your/zenai/source"             # checkout root
```

### 3.2 Build + sign the package

```bash
./Doc/ZENAI/build-zenai-release.sh 1.0.0
# → /tmp/zenai-update-1.0.0.zup
```

The script:
1. Copies your source tree into a staging dir.
2. Writes a `manifest.json` with `"product": "zenai"` and a `steps[]` list the appliance agent will execute (stop services → backup → apply code → pip install → start services → health check).
3. Generates `checksums.sha256` over every file in the package.
4. Signs `manifest.json` with your Ed25519 key → `manifest.json.sig` (raw 64-byte signature).
5. Re-verifies the signature locally with the matching public key.
6. Tars the whole thing into a `.zup`.

Inspect the manifest before uploading — the default `steps[]` in the script is a sensible starting point, but you'll almost certainly want to customise the services list, health-check URL, and migration paths for your actual ZenAI app.

### 3.3 Verify locally, before touching the server

```bash
ZUP=/tmp/zenai-update-1.0.0.zup
WORK=$(mktemp -d)
tar -xzf "$ZUP" -C "$WORK"
jq . "$WORK/manifest.json" >/dev/null && echo "OK manifest parses"
[[ "$(stat -c%s "$WORK/manifest.json.sig")" == "64" ]] && echo "OK signature size"
( cd "$WORK" && sha256sum -c checksums.sha256 --quiet ) && echo "OK file checksums"
sha256sum "$ZUP"    # note this value; the server will return the same
rm -rf "$WORK"
```

If any of those fails, fix the build — **do not** upload a half-broken package.

### 3.4 Authenticate

```bash
read -rsp "Admin password: " ADMIN_PASS; echo
JWT=$(curl -fsS -X POST "${ZENTRYC_API}/admin/auth/login" \
  -H "Content-Type: application/json" \
  -d "{\"email\":\"${ZENTRYC_ADMIN_EMAIL}\",\"password\":\"${ADMIN_PASS}\"}" \
  | jq -r .token)
unset ADMIN_PASS
[[ -n "$JWT" && "$JWT" != "null" ]] || { echo "Login failed"; exit 1; }
AUTH=(-H "Authorization: Bearer ${JWT}")
```

Never write `$JWT` to a file. JWT expires in 24 h.

### 3.5 Upload — **this is the single ZenAI-specific step**

```bash
curl -fsS -X POST "${ZENTRYC_API}/admin/releases/create" \
  "${AUTH[@]}" \
  -F "file=@/tmp/zenai-update-1.0.0.zup" \
  -F "version=1.0.0" \
  -F "product=zenai" \
  -F "changelog=Initial ZenAI release" \
  -F "severity=normal" \
  -F "min_version=1.0.0" \
  | tee /tmp/zenai-release.json

RELEASE_ID=$(jq -r .id /tmp/zenai-release.json)
echo "Release ID: $RELEASE_ID"
```

The `product=zenai` field is the key — without it the release defaults to `ota` (ZenPlus) and would be served to the wrong appliances.

Successful response:

```json
{
  "id": "...",
  "product": "zenai",
  "version": "1.0.0",
  "package_sha256": "...",
  "package_size": ...
}
```

Errors you may see:

| HTTP | Meaning | Fix |
|---|---|---|
| `400 product must be one of ['ota', 'zenai']` | Typo in the `product` field | Use exactly `zenai` |
| `400 Invalid tar.gz file` | Corrupt archive | Re-build |
| `400 Package missing manifest.json` / `.sig` | Wrong archive layout | Confirm `manifest.json` is at archive root |
| `409 release zenai:v1.0.0 already exists` | Same version already uploaded in this channel | Bump to `1.0.1` |
| `401 / 403` | JWT expired or no permission | Re-authenticate, or ask OTA admin for zenai permission |

### 3.6 Publish

```bash
curl -fsS -X POST "${ZENTRYC_API}/admin/releases/${RELEASE_ID}/publish" "${AUTH[@]}"
```

Published releases are visible to the rollout engine, but nothing ships until there's a rollout policy.

### 3.7 Staged rollout — always start with canary

```bash
# Stage 1 — canary (internal lab appliances tagged rollout_group="canary")
curl -fsS -X POST "${ZENTRYC_API}/admin/rollouts" "${AUTH[@]}" \
  -H "Content-Type: application/json" \
  -d "{
    \"release_id\":     \"${RELEASE_ID}\",
    \"stage\":          \"canary\",
    \"target_group\":   \"canary\",
    \"target_pct\":     100,
    \"auto_promote\":   false,
    \"promote_after\":  \"24 hours\",
    \"max_failure_pct\": 5
  }" | tee /tmp/zenai-rollout.json

ROLLOUT_ID=$(jq -r .id /tmp/zenai-rollout.json)

# Monitor
curl -fsS "${ZENTRYC_API}/admin/rollouts/${ROLLOUT_ID}/status" "${AUTH[@]}" | jq .

# Stage 2 — promote to percentage (say 25% of "stable" group)
curl -fsS -X PATCH "${ZENTRYC_API}/admin/rollouts/${ROLLOUT_ID}" "${AUTH[@]}" \
  -H "Content-Type: application/json" -d '{"action":"promote"}'

# Stage 3 — full (100% of stable)
curl -fsS -X PATCH "${ZENTRYC_API}/admin/rollouts/${ROLLOUT_ID}" "${AUTH[@]}" \
  -H "Content-Type: application/json" -d '{"action":"promote"}'
```

Abort or pause if canary fails:

```bash
curl -fsS -X PATCH "${ZENTRYC_API}/admin/rollouts/${ROLLOUT_ID}" "${AUTH[@]}" \
  -H "Content-Type: application/json" -d '{"action":"abort"}'
```

### 3.8 Confirm in the dashboard

Log in at `https://zentryc.com/ota/login/` → sidebar → **ZenAI Releases** → click your release. You'll see the same rollout progress panel as the ZenPlus view, scoped to ZenAI only.

## 4. PULL — what the appliance agent does

The agent is a long-running systemd service on every ZenAI appliance. It **registers once**, then **loops** forever: check in every ~15 min, download/verify/apply any offered release, report the outcome. It never has admin credentials — only its own `api_key`, issued at registration.

### 4.1 Filesystem layout on the appliance

```
/opt/zenai/updater/
├── agent                     # your updater binary/script (you build this)
├── keys/
│   └── zentryc-zenai.pub     # your Ed25519 public key, baked into the image
├── state.json                # {appliance_id, api_key, current_version, last_checkin}
└── logs/updater.log
```

Mode `0700` on the dir; `0600` on `state.json`; owned by a dedicated `zenai-updater` user.

### 4.2 First boot — register once

```bash
REG_TOKEN="a1b2c3d4e5f6..."    # 48-char hex, from the OTA admin
HOSTNAME=$(hostname)

RESP=$(curl -fsS -X POST https://zentryc.com/api/v1/appliances/register \
  -H "Content-Type: application/json" \
  -d "{
    \"registration_token\": \"${REG_TOKEN}\",
    \"hostname\":           \"${HOSTNAME}\",
    \"arch\":               \"amd64\",
    \"os_version\":         \"Ubuntu 22.04\",
    \"current_version\":    \"0.0.0\"
  }")

APPLIANCE_ID=$(echo "$RESP" | jq -r .appliance_id)
API_KEY=$(echo "$RESP"      | jq -r .api_key)

# Persist — the api_key is returned exactly once, store it securely.
jq -n --arg id "$APPLIANCE_ID" --arg k "$API_KEY" --arg v "0.0.0" \
  '{appliance_id:$id, api_key:$k, current_version:$v}' \
  | sudo -u zenai-updater tee /opt/zenai/updater/state.json >/dev/null
sudo chmod 600 /opt/zenai/updater/state.json
```

The response also carries `subscription.license.max_nodes` — cache it. Your agent must refuse to activate more nodes than that value, per `Doc/ZENAI/zenai-appliance-integration.md` §2.

### 4.3 Steady state — check in every 15 min

```bash
APPLIANCE_ID=$(jq -r .appliance_id /opt/zenai/updater/state.json)
API_KEY=$(jq     -r .api_key       /opt/zenai/updater/state.json)
CURRENT=$(jq     -r .current_version /opt/zenai/updater/state.json)

RESP=$(curl -fsS -X POST https://zentryc.com/api/v1/appliances/checkin \
  -H "X-Appliance-ID: ${APPLIANCE_ID}" \
  -H "Authorization: Bearer ${API_KEY}" \
  -H "Content-Type: application/json" \
  -d "{
    \"hostname\":        \"$(hostname)\",
    \"current_version\": \"${CURRENT}\",
    \"agent_version\":   \"0.1.0\",
    \"uptime\":          $(awk '{print int($1)}' /proc/uptime),
    \"services_status\": {\"zenai-engine\": \"running\"}
  }")

NEXT=$(echo "$RESP" | jq -r .next_action)
if [[ "$NEXT" != "update" ]]; then
  exit 0    # no work — caches subscription/license and loops
fi
```

If `next_action == "update"`, the response carries a `release` object:

```json
{
  "next_action": "update",
  "release": {
    "id": "...",
    "version": "1.0.0",
    "changelog": "...",
    "severity": "normal",
    "package_url": "https://zentryc.com/api/v1/updates/download/...",
    "package_sha256": "...",
    "manifest_sig": "<base64 Ed25519 signature>",
    "size": 12345678
  },
  "subscription": { "type": "zenai", "license": { "max_nodes": 50, ... }, ... }
}
```

Because the server filters by `subscription.type`, this release will **always** be a `product=zenai` release. If you somehow get a non-zenai release here, it is a bug — abort and page.

### 4.4 Download

```bash
RELEASE_ID=$(echo "$RESP" | jq -r .release.id)
EXPECTED_SHA=$(echo "$RESP" | jq -r .release.package_sha256)
EXPECTED_SIZE=$(echo "$RESP" | jq -r .release.size)
SIG_B64=$(echo "$RESP"      | jq -r .release.manifest_sig)

OUT=/var/lib/zenai/updater/staging/update-${RELEASE_ID}.zup
mkdir -p "$(dirname "$OUT")"

curl -fsSL \
  -H "X-Appliance-ID: ${APPLIANCE_ID}" \
  -H "Authorization: Bearer ${API_KEY}" \
  -o "$OUT" \
  "https://zentryc.com/api/v1/updates/download/${RELEASE_ID}"
```

The endpoint supports HTTP range requests, so your agent can resume partial downloads on flaky links — use `curl -C -` if your script re-runs after a failure.

**The server enforces a cross-channel guard**: if a zenai appliance tries to `GET /updates/download/<ota-release-id>`, the server returns `403 Release does not match appliance channel`. You don't have to check this yourself, but don't swallow the 403 either — that's a corruption signal.

### 4.5 Verify — **never skip**

Three checks, in this order, before you untar anything into place:

```bash
# 1. Size
[[ "$(stat -c%s "$OUT")" == "$EXPECTED_SIZE" ]] || { echo "size mismatch"; exit 2; }

# 2. SHA-256 of the whole .zup
ACTUAL_SHA=$(sha256sum "$OUT" | awk '{print $1}')
[[ "$ACTUAL_SHA" == "$EXPECTED_SHA" ]] || { echo "sha mismatch"; exit 2; }

# 3. Ed25519 signature over manifest.json
STAGING=$(mktemp -d)
tar -xzf "$OUT" -C "$STAGING"
echo "$SIG_B64" | base64 -d > "$STAGING/manifest.json.sig.fromserver"

python3 - <<PY || { echo "signature invalid"; exit 2; }
from cryptography.hazmat.primitives.serialization import load_pem_public_key
from cryptography.exceptions import InvalidSignature
import pathlib, sys
pub = load_pem_public_key(
    pathlib.Path("/opt/zenai/updater/keys/zentryc-zenai.pub").read_bytes()
)
# The raw signature inside the .zup and the base64 one from the server must match
sig_zup = pathlib.Path("$STAGING/manifest.json.sig").read_bytes()
sig_api = pathlib.Path("$STAGING/manifest.json.sig.fromserver").read_bytes()
if sig_zup != sig_api:
    print("server sig != zup sig", file=sys.stderr); sys.exit(1)
data = pathlib.Path("$STAGING/manifest.json").read_bytes()
try:
    pub.verify(sig_zup, data)
except InvalidSignature:
    print("ed25519 verification failed", file=sys.stderr); sys.exit(1)
print("OK signature verified")
PY

# 4. File-level checksums (catches partial extract / tampering of inner files)
( cd "$STAGING" && sha256sum -c checksums.sha256 --quiet ) || { echo "inner checksums failed"; exit 2; }
```

Any failure here: **abort**, don't attempt to apply, and report `status: "failed"` with a clear `error_message` back to the server (§4.8).

### 4.6 Apply

Run the steps listed in `manifest.json.steps[]` in order. The agent must understand (at minimum) these step types:

| Type | Meaning |
|---|---|
| `stop_services`  | `systemctl stop <name>` for each name in `services[]` |
| `backup`         | Snapshot `code` and/or `database` before mutating anything |
| `apply_code`     | Replace the running code tree from the staging dir |
| `pip_install`    | `pip install -r <requirements>` into the venv |
| `install_binary` | Copy a compiled binary into place, `chmod +x` |
| `start_services` | `systemctl start <name>` for each name in `services[]` |
| `health_check`   | `curl` the given URL; fail if non-2xx within `timeout` seconds |

On any step failure, immediately run `manifest.rollback_steps[]` (`restore_backup` → `start_services`) and report `status: "rolled_back"` with which step failed.

### 4.7 Persist new version

On successful apply:

```bash
jq --arg v "$NEW_VERSION" '.current_version = $v' /opt/zenai/updater/state.json > /tmp/s && \
  mv /tmp/s /opt/zenai/updater/state.json
chmod 600 /opt/zenai/updater/state.json
```

### 4.8 Report

Always report — success, failure, or rollback — so the rollout engine knows what happened.

```bash
curl -fsS -X POST https://zentryc.com/api/v1/updates/report \
  -H "X-Appliance-ID: ${APPLIANCE_ID}" \
  -H "Authorization: Bearer ${API_KEY}" \
  -H "Content-Type: application/json" \
  -d "{
    \"release_id\":    \"${RELEASE_ID}\",
    \"status\":        \"success\",
    \"from_version\":  \"${CURRENT}\",
    \"to_version\":    \"${NEW_VERSION}\",
    \"duration_sec\":  ${DURATION},
    \"error_message\": null
  }"
```

`status` values: `success` | `failed` | `rolled_back` | `downloading` | `applying` | `verifying` (the last three are optional progress pings for long updates).

### 4.9 Systemd unit — minimum viable

```ini
# /etc/systemd/system/zenai-updater.service
[Unit]
Description=ZenAI OTA update agent
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
User=zenai-updater
Group=zenai-updater
ExecStart=/opt/zenai/updater/agent
Restart=on-failure
RestartSec=30s
StandardOutput=append:/opt/zenai/updater/logs/updater.log
StandardError=append:/opt/zenai/updater/logs/updater.log

[Install]
WantedBy=multi-user.target
```

The agent itself should loop with `sleep 900` between checkins, or use a systemd timer unit with `OnUnitActiveSec=15min`.

## 5. End-to-end smoke test (do this first)

Before building any real appliance image, prove the full loop in isolation:

1. **OTA admin** creates a ZenAI subscription on `https://zentryc.com/ota/fleet/` (Fleet → Subscriptions → New → type: **ZENAI**), max_appliances=1, max_nodes=5, expires well in the future. Generate a registration key.

2. **You**: hand-write a trivial `.zup` that only stops + starts a dummy `echo.service`:

   ```bash
   # version 0.0.1 — no-op test release
   mkdir -p /tmp/z && cd /tmp/z
   cat > manifest.json <<'JSON'
   {
     "format_version": 2,
     "product": "zenai",
     "update_id": "00000000-0000-0000-0000-000000000001",
     "version": "0.0.1",
     "severity": "normal",
     "arch": "amd64",
     "steps":           [{"type":"health_check","url":"http://127.0.0.1:9/","timeout":1}],
     "rollback_steps":  [{"type":"restore_backup"}]
   }
   JSON
   # sign with your zenai private key (same python snippet as §3)
   # build the tar.gz: tar czf /tmp/zenai-update-0.0.1.zup .
   ```

3. **You**: push it via §3.5 → publish → create a canary rollout with `target_group="canary"`.

4. **You**: stand up a test appliance (VM). Register with §4.2, checkin, receive the release, verify signatures, report a synthetic `success` (skip §4.6 for the smoke test — the goal is the network path).

5. **You + OTA admin**: confirm in `https://zentryc.com/ota/zenai/releases/<id>/` that the rollout shows `success: 1`.

Only after that works should you build the real agent + real release.

## 6. What's not yet enforced (be aware)

- **Server-side signature verification**: today the server stores `manifest_sig` but does not cryptographically verify it on upload. The *appliance* must verify (§4.5) — that is the trust anchor. When we turn on server-side verification, the public key path will be `/home/net/keys/zentryc-zenai.pub` on the zentryc.com box; hand us the file (public part only) so we can enable it.
- **Cross-channel download guard is on** (§4.4), but **cross-channel check-in filtering depends on the subscription being correctly set** — if an appliance registers with an OTA subscription by mistake, it will receive OTA releases. Verify the registration token you use was generated under a `type=zenai` subscription.
- **Single arch**: today every ZenAI release is `arch=amd64`. If you need arm64, raise it with OTA team first — the schema supports it but we haven't exercised it.

## 7. Troubleshooting

| Symptom | Likely cause | Action |
|---|---|---|
| Upload returns `400 product must be one of ['ota', 'zenai']` | Typo in `-F product=...` | Use exactly `zenai` (lowercase) |
| Upload returns `409 release zenai:vX already exists` | Already uploaded that version into this channel | Bump the patch version |
| Appliance check-in returns `{"next_action":"none"}` but you just published | No rollout policy yet, or appliance's `rollout_group` doesn't match `target_group`, or appliance's `current_version` already ≥ release version | Create the rollout; check `appliance.rollout_group` on the Fleet dashboard |
| Download returns `403 Release does not match appliance channel` | Appliance's subscription is not `zenai` | Confirm the registration token came from a ZenAI subscription |
| Download returns `404 Package file not found` | Server stored the file under the wrong dir — shouldn't happen now, but flag it | Check `/home/net/packages/zenai/update-<ver>.zup` exists on the server; report to OTA team |
| Appliance signature verification fails | Wrong `zentryc-zenai.pub` baked into the image, OR release was signed with the wrong key | Compare the SHA-256 of `/opt/zenai/updater/keys/zentryc-zenai.pub` on the appliance with the fingerprint on your build server |
| `jq -r .release.manifest_sig` is empty string | Release was uploaded with a zero-byte signature file — broken build | Re-build; the build script in §3.2 refuses to finish if the signature doesn't verify locally |
| Rollout shows `failure_rate` above `max_failure_pct` and auto-aborted | Your apply steps or health-check are broken | Inspect per-appliance `update_history` on the release detail page for `error_message`; fix, bump to next patch version, re-push |

## 8. Checklist — ship only when all are green

**Push-side (release engineer):**
- [ ] `zentryc-zenai.key` at `/secure/zentryc-zenai.key` mode 0600, private, one offline backup in the vault.
- [ ] `zentryc-zenai.pub` handed to the OTA admin and baked into the appliance image.
- [ ] `build-zenai-release.sh 0.0.1` produces a `.zup` that verifies locally.
- [ ] Upload with `-F "product=zenai"` succeeds; response JSON shows `"product": "zenai"`.
- [ ] Publish + canary rollout created; visible in `/ota/zenai/releases/`.

**Pull-side (appliance agent):**
- [ ] Agent registers against a ZenAI subscription (response `subscription.type == "zenai"`).
- [ ] `state.json` mode 0600, owned by `zenai-updater`.
- [ ] Check-in returns `next_action: "update"` and a release once the canary rollout is live.
- [ ] Signature verification (§4.5) is wired and **aborts** on mismatch (unit-tested with a deliberately bad signature).
- [ ] SHA-256 verification is wired and aborts on mismatch (unit-tested with a deliberately truncated file).
- [ ] Apply runs `stop_services → backup → apply_code → … → health_check` and rolls back on failure.
- [ ] Report is sent for **every** terminal outcome (success / failed / rolled_back).
- [ ] `current_version` in `state.json` updated only after a successful report.
- [ ] Systemd unit restarts on crash; logs rotate at 14 days.

**Joint:**
